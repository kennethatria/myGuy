## Issue 1: Images Not Loading (404 Errors)

### Problem

Users reported seeing this error when viewing store items:
```
GET http://localhost:8081/uploads/store/5/1767386069_0.png 404 (Not Found)
```

The item loaded successfully, but images returned 404.

### Root Cause

**Docker WORKDIR Mismatch**

The store-service Dockerfile had a critical path mismatch:

**store-service/Dockerfile (OLD - BROKEN):**
```dockerfile
# Final stage
FROM alpine:latest
RUN apk --no-cache add ca-certificates
WORKDIR /root/              # ❌ Working directory set to /root/
COPY --from=builder /app/main .
EXPOSE 8081
CMD ["./main"]
```

**docker-compose.yml:**
```yaml
store-service:
  volumes:
    - store-uploads:/app/uploads  # ✅ Volume mounted to /app/uploads
```

**store-service/cmd/api/main.go:**
```go
// Serve static files for uploaded images
router.Static("/uploads", "./uploads")  // ✅ Serves from ./uploads
```

**store-service/internal/api/handlers/store_handlers.go:**
```go
// Create uploads directory structure
uploadsDir := "./uploads/store"  // ✅ Saves to ./uploads/store
```

### The Problem Flow

1. **Container starts** with `WORKDIR /root/`
2. **Code saves images** to `./uploads` → Resolves to `/root/uploads/` ✅
3. **Code serves files** from `./uploads` → Looks for `/root/uploads/` ✅
4. **Docker volume** mounts to `/app/uploads` ❌
5. **Images saved to** `/root/uploads/` but **volume is at** `/app/uploads`
6. **Files not persisted** and **not accessible** via volume mount
7. **Result:** 404 errors for all images

### The Fix

**store-service/Dockerfile (NEW - FIXED):**
```dockerfile
# Final stage
FROM alpine:latest

RUN apk --no-cache add ca-certificates

WORKDIR /app  # ✅ Changed from /root/ to /app

# Copy the binary from builder
COPY --from=builder /app/main .

# Create uploads directory with proper permissions
RUN mkdir -p /app/uploads/store && chmod -R 755 /app/uploads

# Expose port
EXPOSE 8081

CMD ["./main"]
```

### Changes Made

| File | Line | Change |
|------|------|--------|
| `store-service/Dockerfile` | 19 | Changed `WORKDIR /root/` → `WORKDIR /app` |
| `store-service/Dockerfile` | 24-25 | Added `RUN mkdir -p /app/uploads/store && chmod -R 755 /app/uploads` |

### Why This Fixes It

1. **Container starts** with `WORKDIR /app` ✅
2. **Code saves images** to `./uploads` → Resolves to `/app/uploads/` ✅
3. **Code serves files** from `./uploads` → Looks for `/app/uploads/` ✅
4. **Docker volume** mounts to `/app/uploads` ✅
5. **Images saved to** `/app/uploads/` and **volume is at** `/app/uploads` ✅
6. **Files persisted** in Docker volume and **accessible** ✅
7. **Result:** Images load successfully! 🎉

### Testing


**After Fix:**

## Issue 2: Cannot Send Messages (JavaScript Error)

### Problem

Users reported this error when trying to send messages:
```
Uncaught (in promise) ReferenceError: canSendMessage is not defined
    at sendMessage (StoreItemView.vue:764:60)
```

### Root Cause

**Leftover Reference from Message Limits Removal**

During the message limits removal work (documented in `frontend-limits-removal.md`), we removed the `canSendMessage` computed property from `StoreItemView.vue` but **missed removing one reference** to it.

**frontend/src/views/store/StoreItemView.vue (BROKEN):**
```javascript
// Line 764 (OLD)
async function sendMessage() {
  if (!newMessage.value.trim() || sendingMessage.value || !canSendMessage.value) {
    //                                                      ^^^^^^^^^^^^^^^^^^
    //                                                      ❌ This variable doesn't exist anymore!
    return;
  }
  // ... rest of function
}
```

The `canSendMessage` computed property was removed in the message limits cleanup, but this check remained.

### The Fix

**frontend/src/views/store/StoreItemView.vue (FIXED):**
```javascript
// Line 764 (NEW)
async function sendMessage() {
  if (!newMessage.value.trim() || sendingMessage.value) {
    //                                                      ✅ Removed !canSendMessage.value check
    return;
  }
  // ... rest of function
}
```

### Changes Made

| File | Line | Change |
|------|------|--------|
| `frontend/src/views/store/StoreItemView.vue` | 764 | Removed `!canSendMessage.value` from condition |

### Why This Fixes It

1. **Before:** Function checked undefined variable → JavaScript error → Message not sent ❌
2. **After:** Function only checks if message is empty or already sending → Works correctly ✅
3. **Consistent with message limits removal** - No limits, no need to check if sending is allowed

3. **Run TypeScript/ESLint checks** before committing
4. **Test in browser** after refactoring

## Technical Details

### Docker Volume Mounts

**How Docker Volume Mounts Work:**
```yaml
volumes:
  - store-uploads:/app/uploads
  #    ^                ^
  #    |                |
  #    |                └─ Path inside container
  #    └─ Named volume (stored on host)
```

This creates a persistent volume that maps to `/app/uploads` inside the container.

**Container Working Directory:**
- `WORKDIR /app` sets the working directory
- `./uploads` resolves to `/app/uploads` when WORKDIR is `/app`
- `./uploads` resolves to `/root/uploads` when WORKDIR is `/root/`

### File Save & Serve Flow

```
User uploads image
       ↓
Frontend sends to: POST /api/v1/items
       ↓
store_handlers.go:104-109
  uploadsDir := "./uploads/store"
  userDir := filepath.Join(uploadsDir, strconv.FormatUint(uint64(userID), 10))
  os.MkdirAll(userDir, 0755)
       ↓
File saved to: ./uploads/store/{userId}/{timestamp}_{index}.{ext}
  With WORKDIR=/app → /app/uploads/store/5/1767386069_0.png ✅
  With WORKDIR=/root → /root/uploads/store/5/1767386069_0.png ❌
       ↓
URL stored in DB: /uploads/store/5/1767386069_0.png
       ↓
main.go:64 serves static files:
  router.Static("/uploads", "./uploads")
  Maps /uploads/* to ./uploads/*
  With WORKDIR=/app → Looks in /app/uploads/ ✅
  With WORKDIR=/root → Looks in /root/uploads/ ❌
       ↓
Docker volume mounted at: /app/uploads
  With WORKDIR=/app → Files accessible ✅
  With WORKDIR=/root → Files not in volume ❌
```
