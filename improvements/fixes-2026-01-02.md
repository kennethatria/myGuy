# Fixes and Improvements - January 2, 2026

## Issues Resolved ✅

### 1. Chat Service Docker Permission Error
**Issue:** `chmod: /app/docker-entrypoint.sh: Operation not permitted`

**Root Cause:** Dockerfile was trying to run `chmod` after switching to non-root user (`USER nodejs`)

**Solution Applied:**
- Moved `COPY docker-entrypoint.sh` and `chmod +x` commands to run BEFORE `USER nodejs` line
- File: `chat-websocket-service/Dockerfile`

**Status:** ✅ Fixed

---

### 2. Frontend Dev Server Startup Error
**Issue:** `TypeError: localStorage.getItem is not a function` when running `npm run dev`

**Root Cause:** `vite-plugin-vue-devtools` was trying to access browser APIs during Node.js build phase

**Solution Applied:**
- Disabled `vueDevTools()` plugin in `vite.config.ts`
- Vue DevTools browser extension still works fine
- File: `frontend/vite.config.ts`

**Status:** ✅ Fixed

---

### 3. Backend Database Migration Conflict
**Issue:** `ERROR: relation "idx_users_username" already exists (SQLSTATE 42P07)`

**Root Cause:** Stale database with conflicting schema from previous runs

**Solution Applied:**
- Reset database with `docker-compose down -v`
- Recreated all services with fresh database
- Command: `docker-compose up -d --build`

**Status:** ✅ Fixed

---

### 4. Chat Service Missing DATABASE_URL
**Issue:** `The DATABASE_URL environment variable is not set`

**Root Cause:** Chat service migration script expected `DATABASE_URL` but docker-compose only provided `DB_CONNECTION`

**Solution Applied:**
- Added `DATABASE_URL` environment variable to chat service in `docker-compose.yml`
- Both variables now provided for compatibility
- File: `docker-compose.yml` (line 52)

**Status:** ✅ Fixed

---

### 5. Frontend API URL Duplication - 404 Errors on Tasks Page
**Issue:** `404 Not Found` error when fetching tasks, URL showing `/api/v1/api/v1/tasks` instead of `/api/v1/tasks`

**Root Cause:**
- Environment variable `VITE_API_URL` already included `/api/v1`
- Code in TaskListView.vue was adding `/api/v1` again
- Result: `http://localhost:8080/api/v1` + `/api/v1/tasks` = duplicate path

**Solution Applied:**
1. Updated `TaskListView.vue` to use endpoint path only (`/tasks` instead of `/api/v1/tasks`)
2. Updated `config.ts` to use environment variables instead of hardcoded URLs
3. Files modified:
   - `frontend/src/views/tasks/TaskListView.vue` (line 381)
   - `frontend/src/config.ts` (lines 2-3)

**Status:** ✅ Fixed

**Before:**
```javascript
// TaskListView.vue
fetch(`${import.meta.env.VITE_API_URL || 'http://localhost:8080'}/api/v1/tasks?...`)
// Result: http://localhost:8080/api/v1/api/v1/tasks ❌

// config.ts
const API_URL = 'http://localhost:8080/api/v1'; // Hardcoded
```

**After:**
```javascript
// TaskListView.vue
fetch(`${import.meta.env.VITE_API_URL || 'http://localhost:8080/api/v1'}/tasks?...`)
// Result: http://localhost:8080/api/v1/tasks ✅

// config.ts
const API_URL = import.meta.env.VITE_API_URL || 'http://localhost:8080/api/v1'; // Uses env
```

**Issue:** `syntax error at or near "BEGIN"` in migration file

**Root Cause:**
- Complex PL/pgSQL DO blocks with nested CREATE FUNCTION statements
- node-pg-migrate doesn't handle explicit transaction management (BEGIN/COMMIT)
- Migration file: `002_store_message_integration.sql`

**Attempted Fixes:**
1. ✅ Created base schema migration (`000_initial_schema.sql`)
2. ✅ Removed explicit BEGIN/COMMIT statements
3. ❌ Still failing on complex DO block syntax

**Current Status:** Chat service (port 8082) is not running

**Impact:**
- Main features work fine (registration, tasks, store)
- Only real-time WebSocket chat messaging is unavailable

**Recommended Solution:**
- Rewrite `002_store_message_integration.sql` to use simpler SQL
- Split complex migration into multiple smaller migrations
- Or use raw SQL execution instead of node-pg-migrate for complex migrations
